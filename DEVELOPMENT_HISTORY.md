# Lasada 開発経緯

## 1. 開発動機：Python エコシステムと LiteLLM への不信
本プロジェクトは、Open-Interpreter 等の既存ツールを導入・維持する際に直面する「環境構築の困難さ」を解決するために開始されました。

*   **LiteLLM 依存の地獄**: 基盤ライブラリである LiteLLM の頻繁な仕様変更により、昨日まで動いていたツールが突然動かなくなる事態が多発。依存関係のバージョン解決に多大な時間を奪われ、本来の開発タスクが阻害される現状がありました。
*   **「隠し仕様」への嫌悪**: URLの自動変換やフィールド名の暗黙的な置換など、ライブラリ側が「気を利かせすぎる」挙動によって、独自のプロキシや Azure 環境下でのトラブルシューティングが極めて困難でした。

## 2. 技術的解決：Rust によるゼロベース再構築
「使い捨てスクリプトではない、永続的に安定した道具」を作るため、Python を完全に捨て、Rust によるフルスクラッチ開発を選択しました。

*   **単一バイナリによる配布**: 巨大な Python 仮想環境を不要にし、コンパイル済みの実行ファイル一つで、どの環境でも即座に、かつ同一の挙動で動くことを目指しました。
*   **通信プロトコルの明示化**: 暗黙の変換をすべて排除し、リクエストの形（JSON）やヘッダーを自律的に探索・固定する「自己適応型エンジン」を自前で実装。これにより、自社プロキシや特殊な API エンドポイントに対しても、コードレベルで確実な接続を確保しました。

## 3. 実装された主要機能
開発の結果、本家 Open-Interpreter に匹敵、あるいは一部で凌駕する機能群を単一の Rust プログラムに統合しました。

*   **マルチ Executor**: Bash, Python, Web (Search/Browse), Computer (GUI操作) を統合。
*   **Local RAG**: `fastembed` を内蔵し、外部 API 不要のローカル・ベクトル検索を実装。
*   **視覚アシスト**: 画面キャプチャ上にグリッド座標を描画し、AI が精密に GUI を操作できる仕組みを導入。
*   **スマート・メモリ**: トークン上限時の自動要約と、ベクトル DB による長期記憶の統合。

## 4. 結論
Lasada は、Python という依存関係の迷宮からエンジニアを解放し、Rust の安定性と自律的な適応能力によって「いつでも、どこでも、確実に動く」ことを追求した、究極の個人用エージェント・コアです。
